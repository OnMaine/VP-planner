# Логика генерации атак (`planStore.generate()`)

## Обзор

Генерация — синхронная функция. При каждом запуске **полностью сбрасывает** предыдущие результаты (`attacks`, `noblePlacements`, `paladinPlacements`, `generationIssues`) и строит новый план с нуля.

---

## Шаг 1 — Пул войск

```
pool[coords] = { ...village.troops }  // shallow copy для каждой деревни
```

Войска из пула **вычитаются** по мере назначения атак. Одни войска не могут уйти дважды. Исходные данные деревни (`village.troops`) не мутируются.

---

## Шаг 2 — Виртуальный бюджет дворян

Считается **один раз** перед циклом по целям.

```
virtualNobleBudget[player] = playerData[player].totalNobles − Σ pool[v].snob  (по деревням игрока)
```

Если у игрока в импорте указано `totalNobles` но дворяне в деревнях не построены — система виртуально достраивает их в подходящую деревню во время генерации. При виртуальном назначении: `pool[v].snob += N`, бюджет уменьшается на N. Деревня попадает в рекомендации по строительству.

## Шаг 3 — Выбор деревни: `pickNobleVillage()`

Единая функция для всех типов атак с дворянами (паровоз, зелёный двор, спам-паровоз). Кандидаты передаются уже отсортированными по дистанции.

Режим выбирается настройкой `MassConfig.noblePriority` (переключатель в MassActivePanel):

### `'distance'` — По дистанции
Ранг кандидата: `distanceIndex × 2` (уже построены) или `distanceIndex × 2 + 1` (виртуальные).

Результат: ближайшая виртуальная **бьёт** дальнюю с готовыми дворянами.

```
Приоритеты:
1. ближайшая деревня, войска есть, дворяне построены
2. ближайшая деревня, войска есть, дворяне виртуальные
3. следующая по дистанции с построенными
4. следующая по дистанции с виртуальными
```

### `'built'` — По имеющимся
Ранг кандидата: `distanceIndex` (уже построены) или `N + distanceIndex` (виртуальные).

Результат: все деревни с готовыми дворянами (ближайшие первые), затем все виртуальные (ближайшие первые).

```
Приоритеты:
1. дворяне построены + ближайшая
2. дворяне построены + дальше
3. виртуальные дворяне + ближайшая
4. виртуальные дворяне + дальше
```

---

## Шаг 4 — Цикл по целям

Для каждой цели из `targets`:

```
byDist        = все деревни, отсортированные по расстоянию до цели
nobleVillages = byDist, отфильтрованные по snobMaxDist
slotArrT      = target.arrivalTime + slot.offsetMs
```

Слоты обрабатываются **по порядку**. Каждый слот = пресет × количество.

---

## Типы пресетов и логика выбора деревни

### 🔴 Паровоз (`train`)

**Кандидаты:** `nobleVillages` (или конкретная деревня если задан `nobleVillageCoords`)

**Выбор через** `pickNobleVillage(trainAttacks.length, candidates, ...)`

Дополнительное условие если поезд содержит `full_off`: `axe + light >= fullOffMinAxe && ram > 0`

**Потребление пула** по типу каждой атаки в поезде:
- `full_off` → берёт всё (axe, light, heavy, ram), snob −1
- `green_off flexible` → axe + light до `greenMax` (≤ 999), snob −1
- `green_off axes/light` → до 999 указанного типа, snob −1
- `split` → половина axe/light/heavy, snob −1
- `half_off` → половина axe/light/heavy + все рамы, snob −1
- `custom_off` → войска по спецификации пресета, snob −1

**Интервал внутри поезда:** `arrT[i] = firstArrT + i * snobIntervalMs`

---

### 🟢 Зелёный двор (`green_off`)

**Кандидаты:** `nobleVillages`

**Выбор через** `pickNobleVillage(1, nobleVillages, target, arrT)`

**Варианты состава:**
- `flexible` → `buildFlexGreen()`: axe + light до `greenMax` (≤ 999), минимум `greenMin`. Без добивки фиксированными юнитами.
- `axes` / `light` → до 999 указанного типа + рамы если `greenWithRams`

---

### 💜 Спам-паровоз (`spam` с `spamTrainSize > 0`)

**Кандидаты:** `nobleVillages`

**Выбор через** `pickNobleVillage(trainSize, nobleVillages, ...)` в цикле `while (trainsLeft > 0)`

Для каждой подходящей деревни генерирует:
- `spamCount` спам-атак (таран/катапульта + пехота до `minAttackSize`)
- `trainSize` спам-дворов с интервалом `snobIntervalMs`

---

### ⚪ Спам (`spam`)

**Кандидаты:** `byDist` (все деревни)

**Условие:** есть хотя бы 1 таран или катапульта, попсайз ≥ `minAttackSize`

Если задано окно (`windowBeforeMin` / `windowAfterMin`) — время прилёта рандомизируется: `[slotArrT − wBefore, slotArrT + wAfter]`

**Из пула вычитается** только таран/катапульта (1 шт). Пехота для добивки до `minAttackSize` **не вычитается**.

---

### 🟠 Офф / Шип / Паладин-офф

**Кандидаты:** `byDist`

Берут указанные войска из пула целиком. Проверяется ночное окно.

---

## Фильтры (реально пропускают деревню)

| Условие | Где |
|---|---|
| Ночное окно отправки (`sendExcludeEnabled`) | `pushAtk()` → `return false` |
| Ночное окно заранее (до мутации пула) | `nightExcludes()` перед выбором |
| Дистанция дворян (`snobMaxDist`) | фильтр `nobleVillages` |
| Недостаток войск | `if (!c)`, `hasArmy()`, `pickNobleVillage` не находит кандидата |

## Предупреждения (только флаг, не фильтруют)

| Код | Условие |
|---|---|
| `SEND_IN_PAST` | `sendTime < now` |
| `NIGHT_ARRIVAL` | прилёт в ночное окно |
| `NIGHT_SEND` | отправка в ночное окно (при `nightActive`) |
| `WATCHTOWER_HIT` | деревня атакующего в радиусе вышки наблюдения |
| `MORALE_HIGH_RISK` | `atkPoints / defPoints < 1.0` |
| `MORALE_MEDIUM` | `atkPoints / defPoints < 1.5` |

## Рекомендации по строительству (`noblePlacements`)

`trackNoble(village)` вызывается каждый раз когда деревня получает дворянина в атаку. Накапливает `{ village, count }`.

Включает **оба случая**:
- Дворяне уже были в пуле (построены) — использованы в атаке
- Дворяне добавлены виртуально из `totalNobles` бюджета — нужно построить

UI сейчас не различает эти два случая.

## Открытие приказов (`openOrdersTo`)

Computed на основе `attacks.value`. Для каждой цели собирает игроков с **реальными** (не спам) атаками. Если два игрока атакуют одну цель реальными атаками — они попадают в список приказов друг другу.

Спам (`type === 'spam'` / `'spam_noble'`) **не учитывается** — спам-игрокам приказы не нужны.

## Механика башни наблюдения (Watchtower)

### Как работает в игре

Башня наблюдения уровня N даёт радиус обнаружения R полей. Соответствие уровня и радиуса задаётся таблицей в настройках мира (`watchtowerRadii`).

Пример: башня 20 уровня → радиус 15 полей.

**Что засвечивается:**
Когда атакующий юнит входит в радиус башни, вражеский игрок видит:
- **Цвет атаки** (зелёный / оранжевый / красный — по кол-ву войск)
- **Наличие дворянина** в атаке

**Отметка сохраняется навсегда** — даже после того как юнит покинул радиус. Это означает: *любое* пересечение радиуса = постоянная засветка этой атаки. Избежать последствий после засвета невозможно.

**Что именно определяет башня:**
Башня игрока X засвечивает:
1. Атаки идущие **в деревни внутри радиуса** башни
2. Атаки идущие **в любые деревни игрока X**, чей маршрут проходит через радиус башни (даже если цель далеко за радиусом)

Башни **союзников** атакующего не засвечивают атаки на чужих игроков.

**Атаки идут по прямой линии** от деревни-источника до деревни-цели. Поэтому задача сводится к проверке пересечения отрезка (маршрут) с окружностью (радиус башни).

### Время до засвета

```
timeUntilDetection = distanceToRadiusEntry / unitSpeed
```

Момент засвета = когда юнит **входит** в радиус. После этого отметка остаётся навсегда.

Хорда влияет не на то произошёл ли засвет, а на **когда** он произошёл и **сколько** времени у защитника до прилёта:

```
timeUntilArrival_after_detection = (distanceFromRadiusExit to target) / unitSpeed
```

Чем позже юнит вошёл в радиус (касательный маршрут) → тем меньше времени у защитника на реакцию.

### Приоритет оптимизации маршрута

```
1. Маршрут не пересекает радиус вообще      → идеально, нет засвета
2. Маршрут пересекает, но хорда минимальна  → засвет поздний, мало времени на реакцию
```

Второй приоритет — минимизация хорды (= минимизация времени нахождения в радиусе = засвет как можно ближе к цели).

### Реализация: `calcWatchtowerExposure(fromX, fromY, toX, toY, targetPlayer)`

Функция определена внутри `generate()` как замыкание над `settings` и `watchtowerVillages`.

**Релевантные башни:** только те, чей `player === targetPlayer` (башня светит атаки на деревни своего владельца). Если `targetPlayer` не задан — проверяются все импортированные башни (консервативный фолбэк).

**Алгоритм:** пересечение отрезка with окружностью через дискриминант; суммирует хорды через все пересекаемые башни.

```ts
// a = segLen², b = 2(f⃗·d⃗), c = f⃗·f⃗ − R²
// disc < 0 → нет пересечения
// tEnter = max(0, t1), tExit = min(1, t2)
// if tEnter >= tExit → пересечение за пределами отрезка
// chord += (tExit - tEnter) * sqrt(segLen²)
```

**Отключение:** если `settings.watchtowerEnabled = false` или нет данных о башнях → функция сразу возвращает `{ detected: false, chord: 0 }`.

### Применение в генерации

Для каждой цели перед циклом по слотам — один прогон по всем деревням:

```ts
for (const v of villages) {
  distToTarget.set(v.coords, calcDistance(...))
  if (watchtowerEnabled) {
    const { detected, chord } = calcWatchtowerExposure(v.x, v.y, target.x, target.y, target.enemyPlayer)
    wtScoreCache.set(v.coords, detected ? 1_000_000 + chord : 0)
  }
}
```

Затем `byDist` сортируется по `wtScore + distance`:

```
score(village) = (detected ? 1_000_000 + chordLength : 0) + distanceToTarget
```

Результат:

```
[без засвета, ближайшие] → [без засвета, дальше] → [засвет мин.хорда] → [засвет макс.хорда]
```

`nobleVillages` наследует этот порядок — `pickNobleVillage` автоматически предпочитает безопасные деревни.

`pushAtk` ставит `WATCHTOWER_HIT` warning по той же `calcWatchtowerExposure` (не влияет на выбор деревни, только информирует).

## Оптимизация маршрутов: итоговая архитектура

### Геометрия: пересечение отрезка с окружностью

```
// Маршрут: точка A (деревня) → точка B (цель)
// Башня: центр C, радиус R

// Вектор AB
dx = B.x - A.x,  dy = B.y - A.y

// Вектор AC
fx = A.x - C.x,  fy = A.y - C.y

a = dx*dx + dy*dy
b = 2*(fx*dx + fy*dy)
c = fx*fx + fy*fy - R*R

discriminant = b*b - 4*a*c

// discriminant < 0  → нет пересечения (маршрут не задевает радиус)
// discriminant >= 0 → есть пересечение, считаем параметры t1, t2
t1 = (-b - sqrt(discriminant)) / (2*a)
t2 = (-b + sqrt(discriminant)) / (2*a)

// Пересечение реальное если хотя бы один t ∈ [0, 1]
intersects = (t1 <= 1 && t1 >= 0) || (t2 <= 1 && t2 >= 0) || (t1 < 0 && t2 > 1)

// Длина хорды (в полях)
chordLength = max(0, min(t2, 1) - max(t1, 0)) * sqrt(a)

// Время засвета
timeInRadius = chordLength / unitSpeedFieldsPerSec
```

### Входные данные для башен

Башни хранятся в `planStore.watchtowerVillages` (персистентно через localStorage `vp_watchtower`).

Поля `WatchtowerVillage`:
- `coords`, `x`, `y` — координаты деревни
- `level` — уровень башни (0–20); уровень = радиус обнаружения в полях
- `player` — владелец (сопоставляется с `target.enemyPlayer` при проверке)

Башни импортируются вручную или через `resolveAllFromMap()` (автоматически проставляет игрока по карте врагов).

## Алгоритмы распределения атак по целям

### Текущая проблема

Сейчас — **жадный алгоритм по порядку целей**: каждая цель в порядке очереди забирает лучшее из пула. Последние цели в списке получают то что осталось — дальние деревни или вообще нехватку.

---

### Вариант 1 — Round-robin (интерлив)

Вместо "все слоты цели 1 → все слоты цели 2" делаем раунды:

```
Раунд 1: Цель1 берёт 1 офф, Цель2 берёт 1 офф, Цель3 берёт 1 офф
Раунд 2: Цель1 берёт 2й офф, Цель2 берёт 2й офф, ...
```

**Плюсы:** просто реализовать, интуитивно справедливо, ближайшие деревни распределяются равномерно.
**Минусы:** не учитывает что одна цель может быть важнее другой, не минимизирует суммарную дистанцию.

**Реализация:** вместо `for target` → `for slot` сделать `for round` → `for target` → взять 1 атаку.

---

### Вариант 2 — Глобальная сортировка пар

Перед генерацией строим список всех пар `(деревня, цель)` с расстояниями, сортируем по дистанции:

```
[(A→T1: 3), (B→T1: 4), (A→T2: 5), (C→T2: 6), (B→T2: 7), ...]
```

Берём по порядку, пропуская уже занятые деревни или цели которые уже набрали нужное кол-во атак.

**Плюсы:** минимизирует суммарную дистанцию по всем целям, оптимальное распределение войск.
**Минусы:** сложнее реализовать, нужно заранее знать сколько атак нужно каждой цели.

---

### Вариант 3 — Венгерский алгоритм (Assignment Problem)

Классическое решение задачи о назначениях: матрица стоимости `cost[village][target]` = расстояние, минимизируем суммарную стоимость назначения.

**Плюсы:** математически оптимально.
**Минусы:** O(n³) сложность, сложная реализация. Для больших масштабов (50+ деревень, 10+ целей) может быть медленным в браузере. Избыточно для TW.

---

### Вариант 4 — Скоринг с учётом башен

Комбинированная метрика для сортировки кандидатов:

```
score(village, target) = 
    distanceToTarget 
  + watchtowerPenalty * timeInWatchtowerRadius
  + (isVirtualNoble ? virtualNoblePenalty : 0)
```

Настраиваемые веса позволяют балансировать между дистанцией и засветом. Интегрируется в `pickNobleVillage` и циклы оффа/спама.

---

### Рекомендуемый подход для TW

**Для оффов и спама** — Вариант 2 (глобальные пары) + учёт башен через скоринг:
- Строим пары (деревня → цель) для каждого слота
- Добавляем штраф за засвет
- Сортируем по итоговому score
- Назначаем жадно, помечая деревни как занятые

**Для паровозов** — Вариант 1 (round-robin по целям):
- Паровоз потребляет всю деревню целиком (дворяне + все войска)
- Жадный round-robin честнее распределяет такие "тяжёлые" ресурсы

**Для спама** — можно оставить текущий жадный, т.к. спам не расходует пехоту. Только тараны расходуются — но их обычно хватает у всех.
