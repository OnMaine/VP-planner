# Логика генерации атак (`planStore.generate()`)

## Обзор

Генерация — синхронная функция. При каждом запуске полностью сбрасывает предыдущие результаты (`attacks`, `noblePlacements`, `paladinPlacements`, `generationIssues`) и строит новый план с нуля.

---

## Шаг 1 — Пул войск

```
pool[coords] = { ...village.troops }  // shallow copy для каждой деревни
```

Войска из пула **вычитаются** по мере назначения атак. Одни войска не могут уйти дважды. Исходные данные деревни (`village.troops`) не мутируются.

---

## Шаг 2 — Виртуальный бюджет дворян

Считается **один раз** перед циклом по целям.

```
virtualNobleBudget[player] = playerData[player].totalNobles − Σ pool[v].snob  (по деревням игрока)
```

Используется когда у игрока в пуле нет построенных дворян, но есть незастроенный лимит. При виртуальном назначении `pool[v].snob += N`, бюджет уменьшается на N.

## Шаг 3 — Цикл по целям

Для каждой цели из `targets`:

### Сортировка деревень

```
byDist      = все деревни, отсортированные по расстоянию до цели (ближайшие первые)
nobleVillages = byDist, отфильтрованные по snobMaxDist
```

### Цикл по слотам масс-конфига

Слоты обрабатываются **по порядку**. Каждый слот = пресет × количество. Время прилёта каждого слота:

```
slotArrT = target.arrivalTime + slot.offsetMs
```

---

## Типы пресетов и логика выбора деревни

### 🔴 Паровоз (`train`)

**Кандидаты:** `nobleVillages` (или `byDist` если задан `nobleVillageCoords`)

**Условия выбора (в порядке приоритета):**
1. `pool[v].snob >= trainAttacks.length`
2. Если в поезде есть `full_off`: `axe + light >= fullOffMinAxe && ram > 0`
3. Отправка не попадает в ночное окно

**Fallback (если не нашли):** `tryVirtualNoble(trainAttacks.length, nobleVillages, ...)`

**Потребление пула:** по типу каждой атаки в поезде (full_off берёт всё, green_off берёт axe/light до 999, split берёт половину, и т.д.)

**Интервал внутри поезда:** `arrT[i] = firstArrT + i * snobIntervalMs`

---

### 🟢 Зелёный двор (`green_off`)

**Кандидаты:** `nobleVillages`

**Условия:**
1. `pool[v].snob >= 1`
2. Отправка не попадает в ночное окно

**Fallback:** `tryVirtualNoble(1, [nv], ...)` — сначала пробует текущую деревню из итерации

**Варианты состава:**
- `flexible` → `buildFlexGreen(a, role)`: берёт axe + light до `greenMax` (≤ 999), минимум `greenMin`
- `axes` / `light` → берёт до 999 указанного типа + рамы если `greenWithRams`

---

### 💜 Спам-паровоз (`spam` с `spamTrainSize > 0`)

**Кандидаты:** `nobleVillages`

**Pre-fill виртуальных дворян:** перед итерацией проходим по всем кандидатам, если `snob < trainSize` и есть бюджет — добавляем виртуально.

**Условия:**
1. `pool[v].snob >= trainSize`
2. Отправка не в ночном окне

Для каждой подходящей деревни генерирует `spamCount` спам-атак + `trainSize` спам-дворов.

---

### ⚪ Спам (`spam`)

**Кандидаты:** `byDist` (все деревни по дистанции)

**Условия:**
- Есть хотя бы 1 таран или катапульта
- Попсайз ≥ `minAttackSize`

Если задано окно (`windowBeforeMin` / `windowAfterMin`) — время прилёта рандомизируется в диапазоне `[slotArrT − wBefore, slotArrT + wAfter]`.

**Пул:** вычитается только таран/катапульта (1 штука). Пехота берётся для добивки до `minAttackSize` но **не вычитается**.

---

### 🟠 Офф / Шип / Палладин-офф

**Кандидаты:** `byDist`

**Условия:** наличие указанных войск в пуле, ночное окно.

Берёт указанные войска из пула полностью.

---

## Ограничения (фильтры, реально пропускающие деревню)

| Условие | Где проверяется |
|---|---|
| Ночное окно отправки (`sendExcludeEnabled`) | `pushAtk()` → `return false` |
| Ночное окно **заранее** (до мутации пула) | `nightExcludes()` перед выбором |
| Дистанция дворян (`snobMaxDist`) | `nobleVillages` фильтр |
| Недостаток войск | `if (!c)`, `if (a.snob <= 0)`, `hasArmy()` |

## Предупреждения (не фильтруют, только флаг в `warnings[]`)

| Код | Условие |
|---|---|
| `SEND_IN_PAST` | `sendTime < now` |
| `NIGHT_ARRIVAL` | прилёт в ночное окно |
| `NIGHT_SEND` | отправка в ночное окно (при `nightActive`) |
| `WATCHTOWER_HIT` | деревня атакующего в радиусе вышки наблюдения |
| `MORALE_HIGH_RISK` | `atkPoints / defPoints < 1.0` |
| `MORALE_MEDIUM` | `atkPoints / defPoints < 1.5` |

## Рекомендации по строительству (`noblePlacements`)

`trackNoble(village)` вызывается каждый раз когда деревня назначается в атаку с дворянином. Накапливает `{ village, count }`.

Включает **оба случая**:
- Дворяне уже были в пуле (построены)
- Дворяне были добавлены виртуально (нужно построить)

UI не различает эти два случая — оба показываются как рекомендации.
